## 执行摘要

本报告介绍了一个多模态个人知识库问答系统的设计、实现、开发过程和评估结果。本项目解决了学生、研究人员以及注重隐私的用户会遇到的一个实际问题：有用的个人知识常常分散在 PDF 文件、扫描图像、截图、笔记和纯文本文件中，而普通文件系统只提供有限的搜索和组织能力。用户可能记得某个概念存在于自己的资料里，但定位相关页面或段落往往缓慢且不可靠。

已实现的系统是一个本地优先的 Web 应用。它允许用户注册、登录、上传文档、从受支持文件中提取文本、创建可检索文本块、检索相关证据、提出自然语言问题，并查看带引用的回答。后端使用 FastAPI、SQLite 和面向服务的 Python 模块实现。前端使用 React、Vite 和 TypeScript 实现。检索流水线围绕本地处理和本地存储设计，并在可选条件下使用 SentenceTransformers 和 FAISS 进行语义检索。当配置了密钥时，回答生成可以使用 DeepSeek 官方 API。如果外部模型不可用，系统仍会基于检索到的文本块返回抽取式回退回答，因此核心工作流仍然可以演示。

本项目从一个较宽泛的提案逐步演化为一个可运行的概念验证系统。最终系统并不试图成为生产级云平台，而是专注于核心软件设计挑战：把文档摄取、解析、索引、检索、回答生成、认证和可用的浏览器界面连接成一个一致的应用。评估表明，系统满足了主要项目目标，同时也暴露了 OCR 安装、大型依赖规模、检索质量、后台处理和更广泛用户测试方面的现实限制。

# 1. 引言与系统概述

## 1.1 所解决的问题

现代学习和专业工作会产生越来越多的个人知识。单个用户可能会积累课堂幻灯片、期刊论文、扫描讲义、项目规范、会议记录、截图、手册和个人总结。这些文件通常包含有价值的信息，但当用户无法快速找到相关部分时，它们的价值就会降低。标准文件系统提供文件夹、文件名、修改日期，有时也提供基础全文搜索。它们通常不支持诸如“哪份文档解释了检索增强生成？”、“我把隐私要求保存在哪里？”或“哪一页讨论了评估标准？”这样的问题。

当文档是多模态时，问题会变得更难。一个 PDF 可能包含嵌入文本、扫描页面、图表、表格和参考信息。一张图片可能包含印刷文字或拍摄的笔记。用户可能并不记得某个事实来自 PDF、截图还是文本笔记。关键词搜索也有局限，因为用户的问题可能使用了与原始文档不同的措辞。例如，用户可能询问“本地数据保护”，而文档中使用的是“隐私保护型本地存储”。这就产生了一个需求：系统需要能够从异构文件中提取信息，将其组织成可搜索单元，检索最相关内容，并基于用户自己的文档生成回答。

隐私也是一个重要考虑因素。许多现有 AI 知识工具要求用户把源文档上传到云服务。这对公开信息可能可以接受，但对于私人笔记、课程作业、研究草稿、财务记录或敏感个人文件则不一定合适。因此，本项目采用本地优先方法。原始文件、提取文本、元数据和检索索引都存储在本地。只有一小组选中的证据文本块会发送给外部语言模型，并且只有在用户明确配置 API 密钥时才会发送。

## 1.2 项目目标

本项目的目标是构建一个概念验证级的多模态个人知识库问答系统，展示从文档上传到带引用回答的完整流程。系统的目标是易于理解、可演示且可扩展，而不是达到商业规模下的功能完备。

主要目标如下：

1. 提供用于用户登录、文档上传、文档管理和问答的 Web 界面。
2. 在本地存储用户文档和元数据，以支持隐私和可重复的本地演示。
3. 尽可能从纯文本文件、PDF 和图片中提取文本。
4. 将提取出的文本切分为可检索、可引用的文本块。
5. 支持针对用户问题的本地相关文本块检索。
6. 在可用时使用可配置的 DeepSeek 模型生成回答。
7. 当未配置外部模型时，基于检索文本块提供回退回答。
8. 展示包括文档名、页码、文本预览和相关性分数在内的引用。
9. 保持架构模块化，使解析、检索、生成和用户界面可以独立演进。

这些目标与评估要求一致：最终报告需要覆盖整个项目的问题、设计、实现、开发过程、评估和未来工作。

## 1.3 目标用户与运行环境

目标用户是维护学习或工作资料集合，并希望在不把完整资料集合上传到第三方知识平台的情况下提出问题的个人。最直接的用户包括复习课程材料的学生、阅读论文的研究人员，以及管理项目笔记的开发者或分析人员。运行环境是用于课程演示的本地机器。用户运行后端服务器，运行前端开发服务器或生产构建，在浏览器中打开界面，并在本地与应用交互。

由于系统面向课程项目，设计优先考虑清晰行为，而不是高级基础设施。它不假设分布式部署、多服务器、云对象存储或企业身份管理。系统使用本地文件存储上传文档，使用 SQLite 存储元数据，并在前端和后端之间使用简单 API 边界。这使系统更容易检查、运行和评估。

## 1.4 形成的系统概述

已实现的系统包含两个主要应用：后端 API 和前端 Web 界面。后端公开健康检查、认证、文档管理和问答端点。它管理用户记录、密码哈希、基于 JWT 的访问控制、文件存储、文档状态、文本提取、文本块创建、检索和回答生成。前端提供面向用户的工作流：登录、文档上传、文档列表、文档状态显示、重新处理、删除、聊天输入、回答输出和引用展示。

主要用户工作流如下：

1. 用户注册或登录。
2. 用户上传 PDF、文本文件或图片。
3. 后端在本地存储文件并记录文档元数据。
4. 后端提取文本、清洗文本、创建文本块并存储文本块元数据。
5. 用户在聊天界面提出问题。
6. 后端检索属于该用户的最相关文本块。
7. 如果配置了 API 密钥，后端将问题和选中的文本块发送给 DeepSeek。
8. 后端返回回答和引用。
9. 前端展示回答、源文件、页码、预览和分数。

该工作流展示了检索增强生成的核心概念：回答应当基于检索到的证据，而不是只由模型的一般知识生成。因此，引用展示不是可有可无的装饰，而是核心功能，因为它允许用户检查回答来源并判断是否可信。

## 1.5 范围

项目范围被有意限制为概念验证系统。已实现的应用支持演示所需的主要流程，但并不包含生产级知识管理系统中预期的所有功能。

范围内内容：

5. 具有文档 id、页码、文本块索引和文本元数据的切块。
6. 本地检索和引用生成。
7. 可选 DeepSeek 回答生成以及回退行为。
8. 基础后端自动化测试和前端构建验证。

范围外内容：

# 2. 系统设计

## 2.1 设计目标

系统设计由五个目标引导。第一，系统应当本地优先。原始文档应保留在用户机器上，并且在没有外部语言模型时系统仍应有用。第二，系统应当模块化。文档解析、切块、检索、生成、认证和展示应当分离，使一个区域可以改进而不需要重写整个应用。第三，系统应当透明。回答应包含引用，文档状态应可见。第四，系统应足够简单，能够在课程环境中运行。所选架构应避免不必要的基础设施。第五，系统应当可扩展。概念验证应为后台任务、更多文档格式、更好排序和更丰富查看器留下清晰路径。

这些目标导向了客户端-服务器架构。前端负责交互和展示。后端负责可信数据操作、文档处理、检索和与语言模型通信。数据库存储结构化元数据。文件系统存储上传文档和派生的本地产物。

## 2.2 高层架构

仓库组织为一个小型 monorepo。Web 应用位于 `apps/web`，后端位于 `apps/api`，本地运行数据位于 `data`。文档和报告位于 `docs`。

```text
multimodal-kb-qa
  apps/
    web/        React + Vite + TypeScript 前端
    api/        FastAPI 后端和服务
  data/
    uploads/    本地上传文件
    extracted/  提取文本缓存
    indexes/    本地检索索引
  docs/         架构、API 契约和报告
  tools/        工具脚本
```

运行时，浏览器通过 HTTP 与 FastAPI 后端通信。后端读写 SQLite 记录，将文件存储到本地数据目录，通过服务模块处理文档，检索文本块，并可选调用 DeepSeek API 进行生成。设计保持了小而稳定的公开接口：

| 区域 | 职责 |
| --- | --- |
| 前端 | 用户工作流、上传控件、文档列表、聊天界面、引用展示 |
| API 路由 | HTTP 端点、请求验证、认证依赖、响应格式 |
| 服务 | 存储、文档处理、切块、检索、LLM 调用、RAG 编排 |
| 数据库 | 用户、文档、文本块、状态、所有权元数据 |
| 本地数据文件夹 | 上传原件、提取文本、索引、运行时产物 |
| 外部 LLM | 基于选中的检索文本块进行可选回答生成 |

这种分离降低了耦合。例如，前端不需要知道检索使用 FAISS 还是回退词法方法。它只调用 `POST /qa/query`，并渲染后端返回的回答和引用。

## 2.3 后端组件设计

后端分为路由、核心配置、数据库模型和服务。路由定义公开 API。服务实现业务行为。这种设计使端点函数保持简短，也让处理流水线更容易测试。

主要后端组件如下：

| 组件 | 职责 |
| --- | --- |
| `main.py` | 创建 FastAPI 应用，配置 CORS，引入路由，并公开健康检查 |
| `core/config.py` | 加载数据库路径、数据目录、JWT 设置和 DeepSeek 设置等环境变量 |
| `core/security.py` | 处理密码哈希、令牌创建和令牌验证 |
| `db/models.py` | 定义用户、文档和文本块的 SQLModel 实体 |
| `db/session.py` | 创建 SQLite 引擎和数据库会话 |
| `routers/auth.py` | 实现注册、登录和当前用户查询 |
| `routers/documents.py` | 实现上传、列表、详情、删除和重新处理端点 |
| `routers/qa.py` | 实现问答端点 |
| `services/storage_service.py` | 保存上传文件并解析本地文件路径 |
| `services/document_processor.py` | 从受支持文件格式中提取文本 |
| `services/chunking_service.py` | 清洗提取文本并切分为文本块 |
| `services/vector_service.py` | 存储文本块元数据并检索相关文本块 |
| `services/llm_service.py` | 在配置可用时调用 DeepSeek chat completion API |
| `services/rag_service.py` | 编排检索、提示词构建、回答生成和回退响应 |

面向服务的设计提供了清晰的职责边界。如果 PDF 提取失败，问题属于处理器。如果回答质量较弱，可以改进检索和 RAG 服务，而无需改变认证或文档上传。如果前端布局变化，后端契约可以保持不变。

## 2.4 前端组件设计

前端是一个包含页面和可复用组件的 React 应用。它有意保持简单，因为项目的主要价值在于从文档到回答的工作流。界面避免营销式落地页，直接进入功能屏幕。

主要前端部分如下：

| 组件或页面 | 职责 |
| --- | --- |
| `LoginPage` | 用户登录和注册流程 |
| `DocumentsPage` | 上传、搜索、文档列表、状态显示、重新处理和删除操作 |
| `ChatPage` | 主要问答屏幕 |
| `UploadPanel` | 文件选择、格式验证和上传状态 |
| `DocumentTable` | 文档行、状态徽标、操作和搜索结果 |
| `ChatWindow` | 消息列表、输入框、加载状态和错误展示 |
| `CitationList` | 展示文件名、页码、预览和分数的引用卡片 |
| `lib/api.ts` | 集中的 API 调用和 TypeScript 类型 |

这种结构使功能区域相互分离。文档管理页面可以独立于聊天页面演进，而二者共享同一个 API 客户端层和认证状态。引用列表作为单独组件存在，因为引用渲染对信任非常关键，未来可以扩展为源预览或文档高亮。

## 2.5 关键交互

上传交互从用户选择受支持文件并提交开始。前端向后端发送 multipart 请求。后端认证用户、保存原始文件、创建文档记录并运行处理。处理期间，文档状态会经过已定义状态：`uploaded`、`processing`、`indexed` 和 `failed`。如果提取和切块成功，文本块会被存储，文档变为可搜索。如果任何处理步骤失败，文档会被标记为失败，并保存错误消息，使前端能够显示可读的失败原因。

问答交互从用户提交问题和可选 `top_k` 值开始。后端认证用户，只从该用户文档中检索相关文本块，构建引用，并要求 RAG 服务生成回答。如果已配置 DeepSeek 且请求成功，则返回生成式回答。如果模型不可用，则后端返回由最相关本地文本块组成的回退回答。在两种情况下，引用都以相同结构返回，因此前端不需要两套不同的渲染路径。

## 2.6 API 契约

公开 API 被有意保持较小：

| 方法 | 端点 | 用途 |
| --- | --- | --- |
| `GET` | `/health` | 检查后端可用性 |
| `POST` | `/auth/register` | 创建用户账号 |
| `POST` | `/auth/login` | 认证并返回 bearer token |
| `GET` | `/auth/me` | 返回当前认证用户 |
| `POST` | `/documents/upload` | 上传并处理文档 |
| `GET` | `/documents` | 列出用户文档 |
| `GET` | `/documents/{document_id}` | 获取单个文档 |
| `DELETE` | `/documents/{document_id}` | 删除单个文档 |
| `POST` | `/documents/{document_id}/reprocess` | 对单个文档重新运行处理 |
| `POST` | `/qa/query` | 提问并返回带引用的回答 |

QA 响应包含一个回答和一组引用。每条引用包括文档 id、文档名、页码、文本块 id、文本预览和分数。稳定的响应结构使前端可预测，并支持未来功能，例如在引用页打开文档查看器。

## 2.7 隐私、安全与设计理由

隐私主要通过本地存储和有限外部披露来处理。上传文件存储在本地数据目录下，并被 Git 忽略。提取文本和索引也属于本地运行时产物。外部模型只接收选中的检索文本块，而不是完整文档集合。这并不能消除所有隐私风险，因为检索文本仍可能包含敏感信息，但它显著缩小了发送到机器外部的内容范围。

安全通过密码哈希、JWT bearer token、路由级认证依赖以及文档和 QA 操作中的所有权检查来处理。每份文档都属于一个用户，检索范围限定在该用户的文档内。这一点很重要，因为即使是本地演示应用，也不应把一个用户上传的材料暴露给另一个账号。

架构使用 FastAPI，因为它提供清晰的类型化 API、自动请求验证和交互式文档。SQLite 被使用是因为它简单、本地化，并且对概念验证足够。React 和 TypeScript 被使用是因为它们支持可维护的交互式前端。本地优先设计也支持课程环境：评估者可以在没有云基础设施的情况下运行系统。

一个重要设计决策是让文档状态显式化，而不是把处理作为不可见的内部动作。四个状态值 `uploaded`、`processing`、`indexed` 和 `failed` 为前端和用户提供了共同词汇，用来说明已经发生了什么。这尤其有用，因为文档处理比普通表单提交更不可预测。文本文件可能立即完成，PDF 可能需要更久，图片如果未安装 OCR 可能失败。显式状态让这些差异变得可理解，并减少测试时的歧义。

另一个重要决策是把引用保留为结构化数据。较松散的设计可能会要求模型在回答文本中包含文件名，但这会让引用质量依赖模型的格式。已实现的设计将引用元数据保留在自由文本回答之外。这对前端更可靠，也为后续功能做好准备，例如打开来源、按文档筛选或引用高亮。

# 3. 实现

## 3.1 技术栈

实现使用了一个实用技术栈，其选择依据是清晰性、本地运行能力和与项目目标的兼容性。

| 层级 | 技术 | 原因 |
| --- | --- | --- |
| 前端 | React、Vite、TypeScript | 快速开发、类型化 UI 代码、简单本地构建 |
| 后端 | FastAPI、Python | 类型化 API 层，对文件上传和服务模块支持良好 |
| 数据库 | SQLite 与 SQLModel | 具有类型模型的本地持久化，设置成本低 |
| 认证 | 密码哈希和 JWT | 受保护 API 路由的标准方法 |
| PDF 处理 | PyMuPDF | 从文本型 PDF 中可靠提取内容 |
| 图片 OCR | pytesseract 和 Tesseract | 常见的本地图片文字识别工具链 |
| 切块 | 自定义服务 | 使文本块元数据与项目引用要求保持一致 |
| 检索 | 本地文本块检索，可选 SentenceTransformers 和 FAISS 路径 | 即使重型 ML 依赖不可用也能演示 |
| LLM 生成 | DeepSeek 官方 API | 在不发送原始文档的情况下进行可配置外部生成 |
| 测试 | Pytest 和前端构建检查 | 验证后端行为和前端编译正确性 |

该技术栈反映了概念验证目标。它避免了重量级部署基础设施，并使大部分行为可在仓库中检查。它也允许回退模式，这在实现期间很重要，因为 OCR 引擎、embedding 模型和网络访问在不同机器上可能不同。

## 3.2 仓库与配置

仓库围绕应用边界组织，而不仅仅按技术层组织。后端代码位于 `apps/api`，前端代码位于 `apps/web`，本地运行数据位于 `data`。这使项目易于浏览，并支持并行开发。

配置从环境变量加载。最重要的运行时变量包括数据库 URL、数据目录、JWT 密钥、令牌过期时间、CORS origins、`DEEPSEEK_API_KEY` 和 `DEEPSEEK_MODEL`。DeepSeek 密钥被有意设计为可选。如果未设置，系统仍然运行并产生回退回答。这让演示更稳健，也避免每个功能都依赖外部账号。

`.env.example` 文件记录了预期变量，但不要求提交密钥。本地 `.env` 文件被 Git 忽略。上传文档、提取文本、索引和本地数据库文件也被忽略，因为它们是运行时数据而不是源代码产物。

## 3.3 后端实现

后端实现遵循分层结构。API 路由验证请求并处理 HTTP 层关注点。服务模块实现实际工作。数据库模型表示持久化数据。

认证从注册开始。用户提交邮箱和密码。后端验证唯一性、哈希密码并存储用户。登录时，后端验证密码并返回 JWT bearer token。受保护路由使用一个依赖项验证令牌并加载当前用户。这使认证逻辑在文档和 QA 端点中保持一致。

文档上传接受 multipart 文件。后端把原始文件保存到本地存储，创建文档记录，并处理该文件。文档状态提供了一个简单状态机。新文档从 `uploaded` 开始，变为 `processing`，当文本块可用时变为 `indexed`，如果提取或索引失败则变为 `failed`。该状态模型在前端可见，也便于调试。

文档处理服务根据文件类型选择提取行为。纯文本文件会被解码和规范化。PDF 使用 PyMuPDF 按页处理，以便保留页码元数据。图片在安装了 Tesseract OCR 引擎时会传入 OCR。若 OCR 不可用，服务会返回明确错误，而不是静默失败。保留页码很重要，因为引用应引导用户回到源位置。

切块服务清洗文本并将其切分为有重叠的文本块。重叠降低了重要句子与其上下文被切开的可能性。每个文本块存储文档 id、页码、文本块索引和文本。这些元数据使检索结果可解释，并允许回答包含有用引用。

向量与检索服务负责存储文本块并寻找相关文本块。架构支持语义 embedding 和 FAISS，同时概念验证实现也支持本地回退检索路径。这是一个实际选择，因为 embedding 模型下载可能很大，并且在评估期间不一定可用。回退路径仍然展示了完整的问题-回答-引用循环。

RAG 服务编排检索和回答生成。它检索 top 文本块，将它们格式化为证据，构造提示词，在可能时调用 DeepSeek 服务，并将最终回答与引用打包。回退回答直接使用检索到的文本块，并明确说明没有生成外部 LLM 回答。这一行为很重要，因为它防止 API 密钥缺失时用户界面看起来像是损坏。

后端还把所有权检查放在需要它们的数据操作附近。列出文档、读取单个文档、删除文档、重新处理和问答都依赖认证用户。这防止意外的跨用户访问，并使行为更容易推理。即使项目主要是本地演示，正确实现所有权仍然是良好系统设计的一部分，因为它防止概念验证依赖不安全假设。

## 3.4 前端实现

前端通过独立页面和组件实现主要用户工作流。登录页面支持注册和登录。应用保存返回的令牌，并将其用于认证 API 请求。错误状态会展示给用户，而不是隐藏在控制台中。

文档页面提供文件选择、上传提交、格式检查、文档列表显示、按名称搜索、状态徽标、删除操作和重新处理操作。状态徽标让处理流水线可见。例如，一个 OCR 失败的文档不应看起来与已索引文档相同。这帮助用户理解文件是否已准备好用于 QA。

聊天页面提供问题输入、加载状态、回答展示和引用。引用展示源文档名、页码、预览和分数。引用组件的设计使未来可以添加展开、打开来源或文档高亮，而无需改变 QA 端点。

前端 API 客户端集中处理请求逻辑和响应类型。这减少了重复，并使前端更容易与后端契约保持一致。TypeScript 在这里很有用，因为 QA 响应具有结构化形状，需要被一致渲染。

视觉设计有意保持克制。系统是一个操作型工具，而不是营销网站。界面优先考虑上传、状态扫描、问题输入和引用阅读。这符合课程演示需求：评估者应能快速理解并使用系统。

前端也体现了系统的可靠性目标。耗时或容易失败的操作具有可见 UI 状态。上传会展示进度或完成反馈。文档会显示其已索引或失败。QA 请求会显示加载和错误状态。这些行为很简单，但很重要，因为检索与生成应用否则可能显得很神秘：用户可能不知道系统是在思考、损坏、等待网络，还是缺少数据。

## 3.5 RAG 提示词与回答生成

生成提示词被设计为强制基于检索证据。后端将用户问题和编号后的检索文本块列表传给模型。模型被指示使用给定上下文回答，并避免没有支持的断言。引用由后端响应以结构化方式处理，而不是只依赖模型在自由文本中产生来源标签。

这种设计有两个好处。第一，它降低幻觉风险，因为模型获得的是狭窄上下文。第二，它将回答文本与引用元数据分离。即使模型措辞变化，前端仍能可靠渲染引用。模型的工作是写出有用回答；后端的工作是保留可追溯性。

当 DeepSeek API 不可用时，回退回答不会被呈现为模型生成的结论。它会明确说明没有生成外部 LLM 回答，然后列出最相关文本块。这使系统保持诚实，并保留有用行为。

提示词设计有意保守。它不要求模型编造缺失信息、推断隐藏上下文，或在检索段落不足时用通用知识回答。这种选择可能使某些回答更短，但更符合个人知识库的目的。用户通常是在询问自己的文档说了什么，而不是请求通用互联网式解释。

## 3.6 错误处理与用户反馈

系统在多个层面处理错误。后端验证错误会作为可读 API 响应返回。认证失败会返回合适状态码。文档处理错误会存储在文档记录上，并通过文档状态展示。OCR 相关失败尤其重要，因为 Tesseract 是外部系统依赖；用户应知道问题是格式不支持、OCR 引擎缺失，还是其他处理错误。

前端在上传和 QA 请求期间展示加载状态。它也展示登录、上传、删除、重新处理和问答失败。这是必要的，因为许多操作比普通 UI 动作耗时更长，尤其是 PDF 提取和模型调用。

## 3.7 设计原则的应用

几个软件设计原则影响了实现。关注点分离体现在路由-服务-数据库拆分中。信息隐藏体现在前端使用 API 函数，而不是直接了解后端内部。单一职责体现在存储、切块和 LLM 调用等服务中。渐进增强体现在可选外部 LLM 和可选语义检索路径中。故障安全行为体现在回退回答路径中。

项目也使用了稳定契约方法。前端依赖端点形状和响应类型，而不是检索如何实现。这允许未来改进后端而不强迫前端大规模修改。同样，文档处理可以增加更多格式，而无需改变 QA 页面。

## 3.8 重大技术挑战

第一个挑战是多模态提取。文本文件很简单，但 PDF 和图片并不简单。PDF 页面可能包含可选择文本、扫描图像或混合内容。图片 OCR 依赖已安装的外部引擎。实现通过清晰处理路径和明确失败状态来处理这一问题。

第二个挑战是在本地优先行为和 LLM 生成之间取得平衡。把完整文档发送给模型会更容易，但会违背隐私目标。最终设计只发送检索文本块，这更可控，也更高效。

第三个挑战是保持演示稳健。外部 API 可能失败，密钥可能缺失，机器学习包可能很重。回退检索和回答路径确保应用即使在受限环境中仍能展示核心概念。

第四个挑战是保留引用。没有引用的 QA 系统听起来有用，但很难信任。因此实现会将文档名、页码、文本块 id、预览和分数贯穿整个流水线。这比简单地把文本发送给模型并显示响应需要更谨慎的数据建模。

## 3.9 测试与运行

后端测试覆盖认证、上传行为、处理、切块和 QA 回退行为。这些测试聚焦最重要的风险：受保护端点应要求认证，上传文件应产生记录，处理应创建可用文本块，QA 应在没有外部模型的情况下仍能工作。

前端验证包括生产构建检查。这确认 TypeScript 类型、导入和组件集成能够成功编译。浏览器工作流仍然需要手动检查，因为上传、文档状态和聊天行为是交互式的。

系统在本地运行。后端从 API 目录启动，前端从 web 目录启动。用户在浏览器中打开前端 URL，注册或登录，上传文档并提出问题。若要生成式回答，用户在本地环境中配置 `DEEPSEEK_API_KEY`。没有该密钥时，回退模式仍然工作。

# 4. 开发过程

## 4.1 开发方法

项目采用迭代式概念验证方法。第一优先级是创建一致架构和稳定 API 契约。第二优先级是实现纵向功能切片：认证、上传、处理、检索、QA 和前端展示。最后优先级是集成、测试、文档和报告准备。

这种方法适合本项目，因为系统跨越多个技术区域。纯前端优先或纯后端优先流程都会延迟集成风险。通过早期定义 API 契约并围绕契约构建功能，项目可以并行推进，同时仍然收敛为一个应用。

## 4.2 迭代策略

工作被组织为多个里程碑：

| 里程碑 | 主要成果 |
| --- | --- |
| 架构搭建 | Monorepo 结构、依赖文件、环境模板、API 契约、文档骨架 |
| 后端基础 | FastAPI 应用、配置、数据库设置、认证、受保护路由 |
| 文档管理 | 上传、存储、元数据、文档状态、列表、删除和重新处理端点 |
| 处理流水线 | 支持格式的文本提取、文本块创建、页码元数据保留 |
| QA 流水线 | 检索、提示词构建、DeepSeek 集成、回退响应、引用 |
| 前端工作流 | 登录、文档管理页面、聊天页面、引用展示 |
| 集成与测试 | 后端测试、前端构建、文档、报告准备 |

每个里程碑都产生一个可工作的增量。这降低了只在最后才发现集成问题的风险，也使系统在整个开发过程中更容易演示。

## 4.3 开发期间的调整

随着实现推进，项目做出了几项调整。第一项是将回退行为保留为一等路径，而不是临时调试辅助。这是必要的，因为外部 LLM 配置应该提升系统，但不应成为每次演示的必需条件。

第二项调整是将模型配置简化为 DeepSeek 官方 API。早期计划允许更通用的 OpenAI-compatible 配置。这种灵活性在设置期间增加了混淆。最终项目使用明确的 `DEEPSEEK_API_KEY` 和 `DEEPSEEK_MODEL` 变量。这对用户更容易理解，也减少了配置错误。

第三项调整是将运行时产物排除在源代码控制之外。上传文档、索引、提取文本和本地数据库文件可能包含个人信息或机器特定状态。将它们加入 Git 忽略规则支持本地优先隐私目标。

第四项调整是优先考虑可见状态和引用。技术上可工作的流水线并不足够，如果用户看不出文档是否失败或回答来自哪里，系统仍然难以使用。因此前端展示文档状态和引用细节。

这些调整说明项目过程不只是实现初始计划。有些设计选择是在团队看到系统运行后变得更清晰。特别是，当前端和后端连接起来后，配置简单性、失败可见性和引用可追溯性变得更加重要。这是软件设计中的常见特征：在提案中看似次要的需求，在用户与可运行系统交互时可能变成核心需求。

## 4.4 团队合作与角色反思

项目可以自然拆分为多个并行角色：前端文档管理、前端聊天、后端认证、后端文档存储、处理流水线、检索与 LLM 集成，以及集成测试。这种拆分与架构相匹配，并减少冲突，因为每个角色都有清晰边界。

最重要的协作机制是 API 契约。一旦端点、请求形状、响应形状和文档状态达成一致，前端和后端工作就可以独立推进。文档也支持协作，因为它记录架构决策和预期行为。

一个关键经验是，并行开发只有在共享边界稳定时才有效。如果 QA 响应形状频繁变化，聊天界面会变得脆弱。如果文档状态值不一致，文档列表会让人困惑。因此项目受益于固定状态值、类型化 API 调用和文档化端点契约。

## 4.5 过程反思

对于概念验证系统来说，该开发过程是有效的，因为它早期聚焦端到端行为。最有价值的进展来自于把上传、处理、检索和回答展示连接成一个流程。一旦该路径可用，单个部分就可以在不丢失整体系统的情况下改进。

该过程的主要弱点是更广泛的评估和用户测试发生较晚。功能测试和构建检查可以为核心行为提供信心，但它们不能完全衡量可用性、回答质量或在多样文档上的性能。未来版本应在流程更早阶段包含一个小型评估数据集和结构化用户任务。

## 5.1 评估方法

系统根据原始项目目标和课程报告预期类别进行评估。评估考虑了功能完整性、架构质量、实现质量、开发过程、限制以及来自测试的证据。由于这是概念验证项目，评估重点在于系统是否展示了预期设计和工作流，而不是它是否已经达到生产就绪。

主要证据形式如下：

1. 已实现的端点和前端工作流。
2. 本地上传、处理、检索和 QA 行为。
3. 文档状态和引用输出。
4. 后端自动化测试。
5. 前端生产构建验证。
6. 对限制和未来改进的清晰记录。

## 5.2 需求满足情况

| 目标 | 结果 |

| 用户认证 | 通过注册、登录、bearer token 和受保护路由实现 |
| 本地文档上传 | 通过后端存储和前端上传 UI 实现 |

## 5.3 验证证据

后端测试验证了重要行为。认证测试检查用户能否注册和登录，以及受保护路由是否要求有效凭证。上传测试检查文件能否被接受和记录。处理与切块测试检查文本能否转化为可检索文本块。QA 回退测试检查当没有外部 LLM 密钥时系统仍然能够响应。

前端构建验证检查 React 应用是否能够成功编译。这可以捕捉损坏的导入、类型不匹配和集成错误。它不能替代手动浏览器测试，但提供了有用基线。

手动验证可以遵循一个简单演示脚本：

1. 启动后端。
2. 启动前端。
3. 注册用户。
4. 上传文本文件、PDF 和图片。
5. 确认成功文件变为 indexed，失败文件展示可读错误。
6. 提出与上传内容相关的问题。
7. 确认回答包含引用。
8. 配置 DeepSeek 并重复问题。
9. 移除或禁用密钥，确认回退模式仍然从本地文本块回答。

该脚本直接反映项目目标，并为功能性和可用性提供证据。

## 5.4 优势

第一项优势是架构清晰。系统在前端、后端、数据库、存储、处理、检索和生成之间具有清晰边界。这使项目更容易理解和维护。

第二项优势是本地优先运行。系统不要求把源文档上传到云知识库。运行时数据保留在本地并被源代码控制忽略。这符合项目的隐私动机。

第三项优势是优雅降级。缺少 DeepSeek 密钥不会破坏整个应用。回退回答路径保持用户工作流可用，也使系统更容易演示。

第四项优势是引用可见性。系统不只是展示回答，还展示来源信息，这提高了信任并支持验证。

第五项优势是适合并行开发。架构自然映射到独立工作区域。这对小组项目很有用，因为团队成员可以负责不同层，而不发生持续文件冲突。

## 5.5 限制

最重要的限制是 OCR 依赖。图片提取依赖机器上正确安装 Tesseract。如果缺失，图片处理会失败。系统会报告失败，但用户仍然需要安装和配置 OCR 支持。

第二项限制是检索质量。回退检索路径对稳健性有用，但语义 embedding 和 FAISS 预期会在改写问题上产生更好结果。在受限环境中，如果措辞差异很大，回退路径可能漏掉相关文本块。

第三项限制是同步处理。大型 PDF 或 OCR 密集型文件可能耗时。生产版本应使用后台任务和进度更新，而不是让处理紧密绑定请求流程。

第四项限制是缺少集成源查看器。引用会显示文件名、页码、预览和分数，但用户还不能点击进入带高亮文本的渲染 PDF 页面。这限制了验证便利性。

第五项限制是评估规模。系统已通过代表性场景测试，但尚未在大型基准集合上或通过结构化用户研究进行评估。

## 5.6 非功能质量评估

可维护性较强，因为代码是模块化的，API 契约明确。后端服务结构使替换或改进单个处理阶段成为可能。前端组件结构将文档管理与聊天分开。

可用性足以满足课程演示。主要工作流可见且直接。然而，生产级可用性需要更好的入门引导、长处理任务的进度指示、更丰富的文档预览以及更清晰的恢复建议。

对于小到中等规模的本地文档集合，性能是可接受的。设计没有针对超大集合优化。未来改进应包括后台处理、增量索引、更好的缓存和分页。

安全性对本地概念验证来说是适当的，但对于生产部署并不完整。密码哈希、JWT 认证和所有权检查已经实现。进一步工作需要速率限制、审计日志、refresh token 轮换、更强密钥管理和加密本地存储。

可靠性通过回退行为和可读失败状态得到提升。然而，OCR 和 LLM API 等外部依赖仍然是失败来源。系统以可演示方式处理这些失败，但生产可靠性需要监控和重试策略。

## 5.7 总体成果

已实现的系统满足了中心项目目标：它展示了一个可工作的多模态个人知识库 QA 流水线。用户可以上传受支持文档，在本地处理它们，提出问题，接收回答，并检查引用。设计一致、模块化，并与本地优先动机一致。

项目也在雄心和可行性之间取得了有用平衡。它包含足够真实的功能而有意义，同时避免了课程概念验证不需要的过度基础设施。主要剩余工作集中在质量、规模和用户体验，而不是基础架构。

从教育角度看，成果也有价值，因为它在一个一致系统中展示了多个软件设计关注点。它包括用户认证、数据所有权、文件处理、外部工具集成、服务边界、类型化前端集成、错误处理和评估。因此结果不只是一个小型 AI 演示，也是一个现代应用功能如何通过严谨软件结构连接起来的现实示例。

# 6. 结论与未来工作

## 6.1 结论

本项目设计并实现了一个本地优先的多模态个人知识库问答系统。所解决的问题是现实的：用户通常拥有许多有用文档，但缺少一种能够在保留隐私的同时跨文档提出自然语言问题的高效方式。最终系统将文档上传、文本提取、切块、检索、可选 DeepSeek 生成、回退回答和引用展示结合为一个可工作的 Web 应用。

主要设计洞察是，一个有用的个人 QA 系统必须既能生成，也能验证。如果用户看不到回答来源，流畅回答并不足够。因此系统把引用视为核心输出。另一个重要洞察是，稳健的本地优先软件应当能够优雅降级。外部语言模型可以提升回答质量，但当外部模型不可用时，系统仍应展示检索和证据。

最终实现满足项目的主要目标，并为未来开发提供了清晰基础。它不是完成的商业产品，但它是一个架构一致、功能可用且限制已记录的概念验证系统。

## 6.2 未来工作

未来工作应关注直接扩展当前设计的改进。第一优先级是更丰富的文档查看器。用户应能够点击引用并打开相关 PDF 页面或图片区域，最好能高亮对应文本。这会大幅加快验证速度。

第二优先级是后台处理。上传大型 PDF 或 OCR 密集型图片时，应创建任务并立即返回。前端随后可以显示进度、排队状态和完成状态。

第三优先级是更强检索。语义 embedding、FAISS 索引、混合词法-语义搜索和 reranking 都可以提高回答相关性。使用小型标注数据集进行评估将有助于衡量这些改进。

第四优先级是更好的本地隐私控制。本地数据库加密、可配置数据目录，以及对发送给外部模型的文本进行更清晰控制，都可以增强用户信任。

第五优先级是更广泛的格式支持。DOCX、PPTX、Markdown、CSV 和 HTML 都是个人知识集合中的现实扩展。每种格式都应尽可能保留来源元数据。

第六优先级是更系统的评估。未来迭代应包括代表性用户任务、回答质量评分、检索准确性检查和可用性反馈。这将推动项目从功能性概念验证走向更成熟的个人知识工具。
