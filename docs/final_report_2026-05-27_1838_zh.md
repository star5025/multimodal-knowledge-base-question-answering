## 执行摘要

本报告介绍了一个多模态个人知识库问答系统的设计、实现、开发过程和评估结果。本项目解决了学生、研究人员以及注重隐私的用户所面临的一个实际问题：个人知识通常分散在
PDF
文件、扫描图像、手写或拍摄的笔记以及纯文本文件中，而传统文件系统通常只提供文件名层面的组织方式和关键词搜索。用户可能知道某个概念存在于自己的资料中，但要定位相关页面、段落或图片往往耗时且不可靠。同时，完全基于云端的知识工具会带来隐私问题，因为原始文档可能需要上传到外部服务器。

本系统实现为一个本地优先的 Web
应用，支持用户认证、文档上传、文档管理、从文本文件和 PDF
中提取文本、在可用时通过 Tesseract 对图片进行
OCR、文本清洗、文本切块、本地元数据存储、检索、问答以及引用展示。后端使用
FastAPI 和 SQLite 实现，前端使用 React、Vite 和 TypeScript
实现。检索层设计为在安装可选机器学习依赖时使用 SentenceTransformers 和
FAISS，同时也包含一个本地词法回退方案，以保证在受限环境中概念验证软件仍然可以演示。当配置了
DeepSeek API 密钥时，系统通过 DeepSeek 官方 API
生成回答。如果没有密钥或外部调用失败，系统会基于检索到的本地文本块返回抽取式回退答案，从而保留端到端演示流程。

本系统从原始项目提案逐步迭代为一个可运行的概念验证系统。实现过程优先考虑架构清晰度、本地数据所有权、可演示性和可追溯输出。系统并不试图成为商业级云平台或企业级知识管理套件。相反，它专注于项目所需的核心流水线：上传、解析、索引、检索、回答和引用。评估表明，已实现的概念验证系统满足项目的核心目标，同时也暴露了围绕
OCR
安装、语义检索依赖规模、回退模式下回答质量以及更广泛可用性测试需求等方面的限制。本报告最后提出了现实可行的未来改进方向，包括更丰富的文档查看器、后台处理、加密本地存储、改进的检索排序以及更完整的评估数据集。

# 1. 引言与系统概述

## 1.1 问题背景

现代学习和专业工作越来越多地产生碎片化的个人知识。单个用户可能会积累课堂讲义、学术论文、扫描材料、项目文档、会议记录、白板照片、技术手册以及个人记录。这些文件往往包含有价值的信息，但当用户无法快速检索相关内容时，它们的价值就会下降。普通文件系统提供文件夹、文件名、日期，有时也提供基础全文搜索。然而，它们通常不提供自然语言接口，使用户能够提出诸如“哪份文档解释了检索增强生成？”、“隐私要求在哪里讨论？”或“哪张扫描笔记提到了评估方法？”这样的问题。

当文档是多模态时，问题会更加复杂。一个 PDF
可能包含嵌入文本、扫描页面、表格、图表或参考文献。一张图片可能包含印刷文字或手写文字。用户可能并不记得某条信息来自
PDF、截图还是文本笔记。关键词搜索通常不足够，因为用户提问时使用的表达可能与文档中的原始用词不同。例如，用户可能询问“本地数据保护”，而文档中使用的是“隐私保护型本地存储”。这推动了一个系统需求：它需要能够从异构文件中提取文本，创建可搜索表示，检索相关内容，并基于用户自己的文档生成回答。

隐私也是一个核心考虑因素。许多现有的 AI
知识库产品依赖云端文档摄取。这可能很有用，但对于个人记录、课程材料、研究笔记、财务文件、医疗信息或私人日记来说并不总是可接受。因此，本项目采用本地优先方法：原始文件、提取内容、元数据和检索索引均存储在本地。只有当用户明确配置
API
密钥时，系统才会将选中的检索文本片段发送给外部大语言模型。这种设计在现代语言模型的实用性和本地存储、本地检索的隐私优势之间取得平衡。

## 1.2 项目目标

本项目的目标是设计并实现一个软件中心的多模态个人知识库问答系统概念验证。系统应展示用户可以上传文档，使其在本地被处理，提出自然语言问题，并收到带有支持性文档片段引用的回答。实现应体现良好的软件设计原则、清晰的架构分离以及实际可行的开发决策。

预期结果并不是一个完整的商业产品。相反，结果是一个技术上可信的概念验证系统，用来验证设计方向。实现应足够稳定，可以用于演示；足够易用，可以让用户在本地操作；并且结构足够清晰，可以支持未来扩展。

## 1.3 目标用户

本项目面向三类主要用户。

第一类是学生，他们需要管理课程讲义、课堂笔记、学术文章、截图和复习材料。学生可能希望询问某个概念出现在哪里、比较不同文档中的解释，或从上传的论文中检索引用。对于这一类用户，重要需求包括可用性、快速上传、简单文档管理和回答可追溯性。

第二类是研究人员和知识工作者，他们需要管理项目文档、会议记录、技术手册、论文和实验记录。他们需要跨文档检索、来源归因，以及无需手动重读大量文档即可提出后续问题的能力。对于这一类用户，引用质量和检索透明性尤为重要。

第三类是注重隐私的用户，他们需要管理不应随意上传到云存储的文档。这些文档可能包括财务记录、医疗报告、合同或私人笔记。对于这一类用户，本地存储、本地检索和受控的外部传输是核心设计约束，而不是可选功能。

## 1.4 已实现系统概述

已实现的系统是一个可通过浏览器访问的本地 Web
应用。它由运行在用户机器上的 React 前端和 FastAPI
后端组成。用户通常通过本地开发地址 `http://127.0.0.1:5173` 访问前端，而
API 运行在 `http://127.0.0.1:8000`。后端将数据存储在本地目录和本地
SQLite 数据库中。

主要功能如下：

| 功能 | 描述 |
|----|----|
| 用户认证 | 用户可以注册、登录，并通过 bearer token 访问受保护的文档和问答端点。 |
| 文件上传 | 用户可以上传 TXT、PDF、PNG、JPG 和 JPEG 文件。不支持的格式会被拒绝。 |
| 文档管理 | 用户可以查看文档名称、类型、大小、处理状态、错误信息和更新时间。 |
| 文本提取 | TXT 文件直接解码，PDF 通过 PyMuPDF 解析，图片在安装 Tesseract 时通过 OCR 处理。 |
| 文本清洗与切块 | 提取的文本会被规范化，并拆分为带有文档和页面元数据的小块。 |
| 检索 | 系统使用可选语义检索和本地词法回退方案检索相关文本块。 |
| 问答 | 后端使用检索文本块构造提示词，并在配置后调用 DeepSeek。 |
| 引用展示 | 回答包含引用，引用中包括文档名、页码、块标识、预览文本和分数。 |
| 优雅回退 | 如果缺少 LLM 密钥或外部调用失败，系统会返回基于本地证据的回答，而不是静默失败。 |

系统运行环境是用于课程演示的本地机器。系统有意设计为不依赖复杂部署基础设施。它不需要云存储、容器编排或第三方身份提供商。

## 1.5 范围

本概念验证实现了项目提案中的核心工作流：用户注册，上传 TXT/PDF/图片文件，后端在本地存储并处理文件，清洗和切分文本，用户提出自然语言问题，系统检索相关片段，并在前端展示带引用的回答。这个端到端路径是本项目的核心评估对象，因为它把文档摄取、多模态文本提取、检索、生成和来源可追溯性连接成了一个可用系统。

项目有意排除了商业云部署、大规模分布式存储、实时协同编辑、原生移动端应用、自训练大语言模型和企业级授权系统。这些排除项使工作量与课程周期匹配，并让项目集中在设计质量、实现可信度和可演示行为上。

# 2. 系统设计

## 2.1 设计目标

系统设计由六个目标引导。

第一个目标是本地优先运行。原始文档应保留在用户机器上。数据库、上传文件和检索数据应存储在本地项目目录中。外部服务只应在用户配置模型
API 密钥之后接收经过检索选中的文本片段。

第二个目标是模块化。文档处理、存储、检索、认证、LLM
生成和前端展示应相互分离。这可以减少耦合，并允许替换系统的部分组件。例如，可以在不重写文档管理
UI 的情况下更换 OCR 后端、embedding 模型或 LLM 提供方。

第三个目标是可追溯性。只有当用户能够验证回答来源时，生成答案才真正有用。因此，系统保留文档名、页码、块标识、预览文本和检索分数。这些引用由
API 返回，并显示在前端。

第四个目标是优雅降级。在学生项目环境中，外部服务、OCR
引擎和较重的机器学习依赖可能并不总是可用。系统在可选组件缺失时仍应能展示核心工作流。因此，检索层包含本地词法回退，回答层包含本地抽取式回退。

第五个目标是部署简单。概念验证系统应易于在开发机器上运行。它使用本地
FastAPI 进程、Vite 前端服务器、SQLite 和文件系统目录。这避免了演示期间对
Docker、托管数据库或远程基础设施的需求。

第六个目标是可扩展性。虽然概念验证系统有意保持有限，但架构应允许现实扩展，包括后台处理、加密存储、更好的语义检索、更丰富的文档预览或更多文件格式。

## 2.2 高层架构

系统采用分层架构。每一层都有清晰职责，并通过明确接口通信。

| 层 | 主要职责 | 主要技术 |
|----|----|----|
| 前端层 | 登录、上传、文档管理、聊天、引用 | React、TypeScript、Vite |
| API 层 | HTTP 端点、认证、请求校验、响应格式化 | FastAPI、Pydantic |
| 持久化层 | 用户、文档和文本块元数据 | SQLite、SQLModel |
| 文件存储层 | 原始上传文件和检索清单 | 本地文件系统 |
| 处理层 | 文本提取、OCR、清洗和切块 | PyMuPDF、pytesseract、Tesseract |
| 检索层 | 文本块搜索和排序 | SentenceTransformers、FAISS、词法回退 |
| 生成层 | 提示词构造和回答生成 | DeepSeek API、回退回答逻辑 |

主要架构流程如下：

1.  浏览器向 FastAPI 后端发送经过认证的请求。
2.  后端验证请求并授权用户。
3.  上传文件存储到 `data/uploads`。
4.  元数据记录到 SQLite。
5.  处理服务提取并清洗文本。
6.  切块服务生成带页面感知的文本块。
7.  检索服务存储文本块元数据，并在问答时检索相关块。
8.  生成服务在配置后只将检索到的文本块发送给 DeepSeek。
9.  后端返回回答和引用。
10. 前端渲染回答和可展开的引用预览。

该架构遵循关注点分离原则。前端不需要知道文本提取或检索如何工作。文档路由不包含底层
OCR 逻辑。LLM 服务不直接操作数据库。每个组件都有聚焦的职责。

## 2.3 后端组件设计

后端实现为 `apps/api/app` 下的 FastAPI
应用。它被组织为路由、服务、数据库模型、配置和安全工具。

路由层暴露公共 API：

| 路由         | 端点                             | 职责                         |
|--------------|----------------------------------|------------------------------|
| 认证路由     | 注册、登录、当前用户             | 账户创建、密码验证、令牌签发 |
| 文档路由     | 上传、列表、获取、删除、重新处理 | 文件管理和处理编排           |
| QA 路由      | 查询端点                         | 问题校验和回答响应           |
| 健康检查端点 | 健康和配置状态                   | 基础运行检查                 |

服务层实现可复用业务逻辑：

| 服务       | 职责                                         |
|------------|----------------------------------------------|
| 存储服务   | 文件类型校验、文件名清洗、上传保存、文件删除 |
| 文档处理器 | TXT 解码、PDF 解析、图片 OCR、文本清洗       |
| 切块服务   | 生成带文档、页面和索引元数据的文本块         |
| 向量服务   | 文本块持久化、检索、语义搜索尝试、词法回退   |
| LLM 服务   | DeepSeek 请求构造和响应提取                  |
| RAG 服务   | 检索编排、引用格式化、回退回答生成           |

数据库模型包含三个主要实体：

| 实体          | 目的                                                   |
|---------------|--------------------------------------------------------|
| User          | 存储用户身份和哈希密码                                 |
| Document      | 存储上传文档元数据、所有者、状态、错误信息和存储文件名 |
| DocumentChunk | 存储提取后的文本块及其所有者、文档、页面和文本         |

设计在文档和文本块记录中使用
`owner_id`，以支持个人知识库。当前概念验证系统通过认证和所有权检查实现基础用户隔离。这不是完整企业级授权模型，但它为私有用户工作区提供了基础。

## 2.4 前端组件设计

前端使用 React 和 TypeScript 实现，位于
`apps/web/src`。它有意聚焦于操作流程，而不是营销式落地页。第一个界面是认证，然后是文档管理和问答。

主要前端模块如下：

| 组件或页面      | 职责                                     |
|-----------------|------------------------------------------|
| `App`           | 认证状态、顶层导航、会话持久化           |
| `LoginPage`     | 注册/登录表单和认证反馈                  |
| `DocumentsPage` | 文档列表、搜索、刷新、删除和重新处理操作 |
| `UploadPanel`   | 文件选择、客户端格式校验、上传状态       |
| `DocumentTable` | 文档元数据、状态徽标、行操作             |
| `ChatPage`      | 聊天状态和 QA 请求处理                   |
| `ChatWindow`    | 消息展示、问题输入、加载状态             |
| `CitationList`  | 可展开引用预览                           |
| `api.ts`        | 后端端点的类型化 HTTP 客户端             |

前端保持界面状态简单。它将认证 token 存储在 local storage
中，在启动时获取当前用户，并在文档视图和聊天视图之间导航。文档视图强调处理状态和错误可见性，这很重要，因为文档处理可能因不支持的文件格式或缺失
OCR 二进制程序等合理原因失败。聊天视图通过引用强调回答可追溯性。

## 2.5 数据流：文档上传

上传流程从用户在浏览器中选择文件开始。前端检查扩展名是否属于支持格式，并将文件作为
multipart form data
发送到后端。后端再次校验文件名，因为客户端校验不足以保证安全性或正确性。

随后后端清洗文件名，并使用生成的唯一存储文件名将文件保存到本地上传目录。这可以避免冲突，并降低不安全路径使用风险。SQLite
中创建文档记录，状态为 `uploaded`，然后开始处理。文档状态被设置为
`processing`，使 UI
能够传达进度。处理成功后，系统写入文本块并将文档状态设置为
`indexed`。如果发生任何错误，该文档的文本块会被删除，状态变为
`failed`，错误信息会被存储以供显示。

这种设计使失败可见且可恢复。失败的文档可以被重新处理。缺失 OCR
引擎、不可读文件、空文档或解析异常不会使应用崩溃，而是成为用户可以检查的文档级失败。

## 2.6 数据流：问答

问答流程从用户在聊天界面提交自然语言问题开始。前端将问题和 `top_k`
值发送到后端。后端验证用户 token，并检索属于该用户的文本块。

如果可选的 SentenceTransformers 和 FAISS
依赖可用，检索服务首先尝试语义检索。它对文档块和问题进行编码，通过向量相似度搜索，并返回排序后的命中结果。如果语义依赖不可用，系统使用基于词项重叠、短语匹配和词项密度的词法评分回退方案。该回退方案在语义能力上较弱，但它保证了在不需要较重机器学习安装的情况下，演示仍然可以运行。

RAG
服务将检索命中转换为提示词上下文。每个命中结果都被编号，并包含源文档名称、页码和文本。LLM
服务向 DeepSeek 官方 chat completions
端点发送系统指令和用户提示词。提示词要求模型只基于提供的摘录回答，在证据不足时说明缺少什么，并使用方括号编号引用来源。如果
DeepSeek 不可用，RAG
服务会从排名靠前的检索块构造本地回退回答，并仍然返回引用。

该流程支持回答可追溯性。模型不会接收整个知识库，而只会接收选中的文本块。前端单独显示引用元数据，因此即使生成答案不完整，用户也可以检查证据。

## 2.7 接口设计

API 契约被设计得较小且稳定，覆盖健康检查、注册、登录、当前用户获取、文档上传、文档列表、文档删除、重新处理和问答。QA 响应同时返回回答文本和结构化引用对象，引用对象包含文档名、页码、片段编号、预览文本和检索分数。把引用作为一等数据比让语言模型在自然语言中自行格式化证据更可靠，也能让前端稳定地渲染引用。

## 2.8 隐私与安全设计

隐私设计包含三层。

第一，原始文件保留在本地。上传文件存储在本地数据目录下，并被 Git
忽略。提取出的文本块和 SQLite
元数据也存储在本地。这可以防止用户文档随源代码被意外提交。

第二，检索在本地进行。系统在任何 LLM
调用之前检索相关文本块。这将外部传输限制为选中的段落，而不是整个文件。同时，当没有配置外部
API 密钥时，系统仍然可用。

第三，认证用于分离用户工作区。密码通过基于 PBKDF2
的方案进行哈希。认证使用 bearer
token。文档和文本块查询限定在当前用户范围内。这适合本地概念验证应用。它并不等同于企业级安全，但展示了访问控制意识，并为更强隔离准备了架构。

安全设计还避免在跟踪文件中存储密钥。DeepSeek API 密钥应放在 `.env`
中，而 `.env` 被 Git 忽略。仓库包含 `.env.example`
来展示所需变量名，但不包含敏感值。

## 2.9 设计理由

为了平衡技术质量和项目可行性，项目做出了若干设计决策。

选择 FastAPI 是因为它提供清晰的请求校验、自动 OpenAPI 文档、良好的
Python 生态兼容性和较快的开发速度。Python 也适合文档处理、OCR
集成和机器学习库。

选择 SQLite
是因为系统是本地优先和单机运行。服务器数据库会增加运行复杂度，但不会改善概念验证。SQLite
足以在本地演示环境中存储用户、文档和文本块。

选择 React 与 Vite
是因为它们提供快速前端开发体验和清晰组件分离。TypeScript 为 API 响应和
UI 状态增加更强的客户端类型检查。

选择 DeepSeek 官方 API 用于回答生成，是因为它提供外部 chat completion
模型，同时使系统独立于云文件存储。当前阶段后端使用单一专用 DeepSeek
集成，而不是通用提供方抽象。这减少了配置混淆，并匹配最新项目决策。

包含回退检索和回退回答路径，是因为概念验证系统必须能够在不完美条件下演示。回退路径也支持在不依赖外部
LLM 的情况下评估本地检索流水线。

# 3. 实现

## 3.1 技术栈

已实现的技术栈如下：

| 领域     | 技术                                                       |
|----------|------------------------------------------------------------|
| 前端     | React、TypeScript、Vite                                    |
| 后端     | FastAPI、Python                                            |
| API 校验 | 通过 FastAPI 使用 Pydantic 模型                            |
| 数据库   | 通过 SQLModel 使用 SQLite                                  |
| 文件存储 | 本地文件系统                                               |
| PDF 处理 | PyMuPDF                                                    |
| 图片 OCR | 通过 pytesseract 使用 Tesseract                            |
| 检索     | 可用时使用 SentenceTransformers 和 FAISS，否则使用词法回退 |
| LLM 集成 | DeepSeek 官方 chat completions API                         |
| 测试     | pytest、FastAPI TestClient、TypeScript build               |
| 环境管理 | 后端使用 uv，前端使用 npm                                  |

该技术栈支持快速实现、本地运行和可扩展性目标。它也使项目对于软件设计与实现评估来说易于理解。

## 3.2 实现组织

实现按后端服务、前端组件、运行时数据、文档和工具进行清晰分离。应用代码与上传文件、提取文本、索引、本地数据库、虚拟环境和密钥分开管理。对于本地优先的知识库系统，这一点很重要，因为运行时文件可能包含私人文档或 API 密钥，不能被误提交到源材料中。

这种组织方式也支持并行开发。只要共享 API 契约保持稳定，后端接口、前端交互、检索实验和文档更新都可以在各自范围内推进。

## 3.3 后端实现

后端应用在启动时加载配置、创建所需数据目录、初始化数据库表并注册路由。应用暴露健康信息和受保护端点。

认证通过本地用户表、密码哈希和 bearer token
实现。这为项目提供访问控制层，而不需要外部身份服务。注册会存储小写邮箱和密码哈希。登录会验证密码并签发签名
token。受保护端点依赖当前用户，从而允许文档和文本块查询按所有者限定范围。

文档路由实现上传和管理生命周期。上传请求校验文件类型并将原始文件存储在本地。处理流程在文档推进过程中更新状态值。明确使用状态值可以使
UI 更清晰，也有助于调试。状态值为 `uploaded`、`processing`、`indexed` 和
`failed`。

文档处理器处理三类文件。文本文件使用一组可能的编码直接解码。PDF 使用
PyMuPDF 逐页解析。图片在系统路径中存在 Tesseract 可执行文件时传给
pytesseract。提取出的文本通过移除空字符、减少重复空格、折叠过多空行并丢弃空行进行清洗。这种简单规范化可以提升文本块质量，而无需构建复杂的自然语言处理流水线。

切块服务将每页文本拆分为可配置大小和重叠量的文本块。切块具有页面感知能力，因此系统可以为引用保留页码。实现还会尽可能尝试在自然边界附近切分，例如换行、句子边界或空格，而不是在任意位置硬切文本。

向量服务将文本块存储到 SQLite，并在问答期间检索它们。如果安装了
SentenceTransformers 和
FAISS，它会尝试语义检索。如果这些包不可用，服务会使用词法评分。这是一个实现层面的设计决策，可以提升鲁棒性。课堂演示不应仅仅因为可选机器学习依赖较重或安装较慢而失败。

LLM 服务有意保持范围狭窄。它调用 DeepSeek 官方 chat completions
端点。请求载荷包含所选模型、系统消息、用户问题、检索到的本地摘录、温度和非流式模式。服务在调用失败时同时返回回答和错误信息。这允许
RAG 层显示有用的回退信息，而不是隐藏原因。

## 3.4 前端实现

前端是一个单页 React
应用。它围绕两个主要认证视图设计：文档管理和问答。UI
功能性强且较为克制，因为该应用是面向工作的工具，而不是公开营销网站。

登录页支持账户创建和登录。一旦收到 token，它会被本地存储并用于 API
请求。应用启动时，前端会调用当前用户端点以检查保存的 token
是否仍然有效。

文档页包含上传面板和文档列表。上传会触发后端处理，返回的文档状态会立即显示。表格显示文件名、文件类型、大小、状态、更新时间和错误信息。用户可以删除文档或重新处理文档。搜索框按文件名过滤，这在知识库包含许多文件时很有用。

聊天页维护本地消息列表。当用户提出问题时，前端将问题发送到 QA
端点并显示返回的回答。如果存在引用，它们会显示在助手回复下方。每个引用都可以展开以显示文本预览。这种设计使用户无需打开原始文档也可以验证回答。

API 客户端集中在单个 TypeScript
文件中。这减少了重复，并将端点路径、请求格式和响应类型集中在一个位置。如果后端契约发生变化，前端修改可以在可预测的位置完成。

## 3.5 RAG 提示词与回答生成

生成提示词有意保持保守。系统提示词要求模型只使用提供的本地知识库摘录回答，在摘录不足时说明缺少什么，并使用方括号编号引用来源。用户提示词包含用户问题和编号摘录。每个摘录包含文档名、页码和文本块内容。

这种提示词设计支持项目的可追溯性要求。它并不能保证完全事实
grounding，因为语言模型仍可能犯错，但它降低了产生无依据回答的机会。单独的结构化引用列表进一步支持验证。

当前提示词不包含查询改写、多步推理指令或回答格式模板等高级功能。这是一个有意的概念验证选择。它让模型交互更容易理解，也使评估更容易。未来工作可以为摘要、比较或学习复习模式加入更复杂的提示词模板。

## 3.6 错误处理

实现包含多种错误处理机制。

不支持的文件格式会被清晰拒绝。空文档或不可读文档会成为失败文档，而不会使后端崩溃。缺失
Tesseract 安装会产生明确 OCR 错误。DeepSeek API
失败会产生本地回退回答，并包含 LLM
错误字符串。认证失败会返回适当错误响应。前端会在可见横幅或文档行中显示错误。

这种方式改善了用户体验和可演示性。在学术项目中，不仅快乐路径正常工作很重要，常见失败情况能否被可预测地处理也很重要。

## 3.7 配置

系统使用环境变量进行本地配置，包括令牌签名密钥、DeepSeek API 密钥、所选 DeepSeek 模型、可选嵌入模型、文本片段长度、片段重叠长度和允许的前端来源。示例环境文件记录所需变量名但不暴露密钥，真实环境文件则被版本控制忽略。这样既能保持设置可复现，也能在课程提交时保护凭据。

## 3.8 设计原则的应用

项目应用了若干设计原则。

关注点分离体现在路由、服务、模型和前端组件之间的划分。存储服务不负责 LLM
调用。LLM 服务不负责数据库查询。前端 API 客户端与 UI 组件分离。

单一职责在模块层面得到应用。每个服务都有清晰目的。这使代码更容易测试和修改。例如，可以在不修改认证路由的情况下改进文档处理器。

优雅降级内置于检索和生成中。语义检索和 DeepSeek
生成在可用时提升质量，但本地回退路径在外部或可选依赖不可用时保留基本功能。

通过 API 契约和类型化前端模型使用显式接口。QA
端点返回结构化引用，而不是要求前端从自然语言中解析引用。

本地优先隐私通过数据位置和处理流程实现。文件存储在本地，检索发生在任何外部请求之前。只有选中的文本块会被发送给
DeepSeek。

## 3.9 重大技术挑战

第一个挑战是在语义检索质量和安装复杂度之间取得平衡。SentenceTransformers
和 FAISS
适合语义检索，但它们会增加大型依赖，并且在受限环境中可能难以安装。已实现的解决方案将这些依赖设为可选，并提供词法回退。这在保持可扩展性的同时，也保证了概念验证可演示。

第二个挑战是现实地处理 OCR。图片 OCR 不仅依赖 Python
库，还依赖系统路径中安装了 Tesseract
可执行文件。系统无法完全控制外部安装。因此实现会检测缺失的 Tesseract
并报告清晰失败。这比静默产生空文本或崩溃更好。

第三个挑战是管理 LLM 提供方配置。开发期间曾考虑使用中间 API
网关，但最终实现直接使用 DeepSeek 官方
API。这降低了配置复杂度，并去除了模糊的 base URL 设置。最终设计只有专用
DeepSeek key 和 model 变量。

第四个挑战是在整个流水线中维护引用元数据。页码和文档名必须经过提取、切块、存储、检索、回答生成和前端渲染后仍然保留。设计通过将引用元数据作为结构化数据与文本块一起存储并单独从生成答案中返回来解决这一问题。

第五个挑战是确保外部 LLM 失败时系统仍然有用。如果模型 API
不可用时整个系统完全失败，那么系统会非常脆弱。回退回答路径提供了较低质量但仍然有信息量的基于检索块的响应。

## 3.10 数据模型与持久化细节

数据模型刻意保持较小，但覆盖所有权、处理状态、检索和引用所需的关系。`User` 实体保存身份与哈希后的认证材料。`Document` 实体保存所有者、原始文件名、生成后的存储文件名、类型、大小、处理状态、错误信息和时间戳。`DocumentChunk` 实体保存所有者、文档、页码、片段序号和片段文本，使检索证据可以追溯到来源。

SQLite 适合本概念验证，因为它本地、可靠、易检查，且不需要独立数据库服务。该设计不面向高并发 SaaS 工作负载，这也属于项目明确排除的范围。运行时文件和数据库不会进入受跟踪的源文件，以降低隐私风险和提交风险。

## 3.11 API 与契约稳定性

API 契约让前端和后端能够相对独立地推进。认证遵循注册、登录、当前用户的模式，文档端点提供上传与管理操作，QA 端点接收问题和检索数量，并返回回答与引用。这种可预测的契约有利于团队协作，因为 UI 工作和后端 RAG 工作可以并行进行。

返回结构化引用是最重要的契约决策。它避免了从生成文本中脆弱地解析来源，也能在回答较短、不完整或由本地回退逻辑生成时保留可见证据。健康检查端点还可以在不暴露 API 密钥的前提下帮助诊断 DeepSeek 是否已配置。

## 3.12 测试策略

测试集中在概念验证风险最高的行为上。后端测试使用 FastAPI 测试客户端，验证注册、登录、令牌使用、文本处理、片段创建、文档上传和 QA 回退行为。前端验证通过生产构建完成，用于检查 TypeScript 编译、模块解析和打包。后端还使用 Ruff 进行静态检查。

该策略与项目阶段相匹配。它为主工作流足够稳定、可以演示提供了证据，同时把更广泛的 OCR 专项测试、检索排序测试、浏览器交互测试和用户评估留作后续工作。

## 3.13 部署与运行

系统以两个本地进程运行：FastAPI 后端和 Vite 前端。该运行模式支持隐私目标，因为文档保留在用户机器上，不需要远程托管。主要前提是后端环境、Node.js 前端环境、用于 OCR 的可选 Tesseract，以及用于生成回答的可选 DeepSeek 密钥。即使没有密钥，系统仍能演示本地检索和引用输出。

对非技术用户而言，运行两个进程只适合作为概念验证。后续版本应提供单一启动器，用于检查依赖、启动两个服务、验证端口并自动打开浏览器。

# 4. 开发过程

## 4.1 方法论

本项目遵循受 Scrum
启发的敏捷迭代开发方法。原始提案将工作划分为前端、后端、算法/RAG、集成和文档职责。实现过程也体现了这一点：系统按功能切片构建，而不是试图孤立地完成每一层。

开发方法优先考虑尽早形成可工作的端到端流程。当核心工作流可以演示时，概念验证最有价值。因此，第一个实现目标不是完整用户界面或完美检索模型，而是最小有意义流水线：注册、上传、提取、切块、提问、检索、回答和引用。一旦该路径存在，设计就可以继续细化，错误处理也可以改进。

## 4.2 迭代策略

开发过程可以描述为六个迭代。

第一轮迭代建立仓库结构和架构。创建了独立的前端、后端、数据和文档目录。添加了
API 契约和架构文档，以保持实现与提案一致。

第二轮迭代实现后端基础。这包括 FastAPI 应用设置、配置加载、SQLite
模型、数据库会话、认证、密码哈希、bearer token
和基础路由。这为受保护工作流建立了基础。

第三轮迭代实现文档摄取。加入文件上传、格式校验、本地存储、元数据创建、状态转换和文档列表。实现了
TXT 和 PDF 文本提取，并通过 pytesseract 和 Tesseract 检测准备了 OCR
支持。

第四轮迭代实现检索和
QA。实现了文本清洗、切块、文本块持久化、检索、DeepSeek
集成、提示词构造、引用格式化和回退回答。这完成了核心 RAG 流水线。

第五轮迭代实现前端。加入登录页、文档管理页、上传面板、文档表格、聊天页、聊天窗口和引用展示。UI
通过类型化 API 客户端连接到后端。

第六轮迭代聚焦集成、测试、文档和配置清理。添加了认证、文档上传、QA
回退、文本处理和切块测试。运行了前端构建和后端 lint。LLM
配置被简化为仅使用 DeepSeek 官方 API 变量。文档被更新为与实现保持一致。

## 4.3 里程碑进展

项目从提案到实现的进展如下：

| 里程碑     | 结果                                                 |
|------------|------------------------------------------------------|
| 需求和提案 | 定义目标用户、本地优先 RAG 范围、技术和排除项        |
| 架构设计   | 建立分层架构和本地数据模型                           |
| 后端基础   | 实现认证、配置、数据库和路由结构                     |
| 文档处理   | 加入文件上传、PDF 解析、文本提取、OCR 路径和状态处理 |
| 检索与 RAG | 加入切块、检索、DeepSeek 生成和回退回答              |
| 前端工作流 | 加入登录、上传、文档列表、聊天和引用 UI              |
| 验证       | 添加自动化测试和构建检查                             |
| 文档       | 创建 API 契约、架构说明、README、工具脚本和最终报告  |

这些里程碑展示了从设计到可运行概念验证的推进。每个里程碑都产生了可用改进，而不仅仅是内部代码变化。

## 4.4 开发过程中的调整

开发过程中进行了若干调整。

原始提案允许使用 OpenAI-compatible 外部 LLM
服务或其他可比提供方。实现期间，项目最初支持可配置的 OpenAI-compatible
base
URL。然而，当使用中间网关时，这产生了配置混淆。随后设计被简化为只使用
DeepSeek 官方 API。这降低了错误 base URL
的风险，并使配置在演示时更清晰。

检索设计也进行了调整。提案将 SentenceTransformers 和 FAISS
列为拟采用技术。实现保留它们作为首选语义检索路径，但提供词法回退。这一调整是必要的，因为可选机器学习依赖可能在受限本地环境中造成安装问题。回退并不替代语义检索作为理想路径，但它保护了概念验证工作流。

测试也根据本地环境进行调整。临时目录和缓存位置被配置在项目内部，以避免权限问题。这使自动检查在开发环境中更可靠。

前端设计也适应了概念验证重点。界面没有构建宽泛仪表盘或落地页，而是聚焦于评估中重要的工作流：上传、查看处理状态、提问和验证引用。

## 4.5 团队合作与角色反思

拟定的团队结构将职责划分为前端、后端、算法/RAG、项目领导和文档。已实现仓库反映了这些边界。前端工作隔离在
`apps/web`；后端 API 和服务工作在 `apps/api`；文档在 `docs`；工具脚本在
`tools`。

这种结构支持并行开发。只要 API
契约保持稳定，前端开发者可以开发聊天界面，同时后端开发者可以开发 QA
端点行为。处理方向的开发者可以改进 OCR 或切块，而不修改登录
UI。文档成员可以更新架构和用户手册材料，而不触碰运行时代码。

开发过程也突出了集成责任的重要性。即使任务被划分，系统价值仍取决于完整工作流。上传、处理、检索、生成和引用展示必须协同工作。因此，集成测试和共享
API 契约对这类项目至关重要。

## 5.4 功能优势

最主要的功能优势是完整的概念验证工作流。系统并未停留在孤立的解析、检索或界面模块，而是把注册、上传、处理、问答和引用展示连接成一个可用应用。引用支持也是重要成果，因为用户可以查看文档名、页码、预览文本和分数，而不是盲目信任无依据回答。本地优先设计和回退行为还使系统在没有云存储、LLM 凭据或可选语义检索依赖时仍然可以演示。

## 5.5 技术优势

从技术上看，系统受益于模块化后端服务、聚焦的前端组件、显式 API 响应、SQLite 持久化和较窄的配置模型。FastAPI 提供请求验证和 API 文档，TypeScript 改善前端契约。检索设计也符合课程环境：依赖可用时支持语义检索，受限机器上则通过词法回退保留基本能力。

## 5.6 限制

第一个限制是 OCR 依赖管理。图片 OCR 需要单独安装 Tesseract
并使其在系统路径中可用。如果缺少
Tesseract，图片处理会清楚地失败。对于概念验证来说可以接受，但在打包产品中应通过安装检查或捆绑设置指南改进。

第二个限制是检索质量。词法回退有用，但无法完全匹配语义检索。它可能检索到具有重叠词语的块，而不是真正语义上最相关的内容。安装
ML extra 可以改善这一点，但完整评估需要基准问题和相关性判断。

第三个限制是同步处理。上传处理当前发生在请求期间。这很简单，但大型 PDF
或缓慢 OCR 可能让用户等待。生产级设计应使用后台任务和进度更新。

第四个限制是文档查看。前端显示引用预览，但尚未提供集成 PDF
或图片查看器，使用户能够直接跳转到引用页面。这限制了来源验证便利性。

第五个限制是安全深度。系统有基础认证和本地用户隔离，但没有实现加密本地存储、基于角色的访问控制、刷新
token、审计日志或强化生产部署设置。

第六个限制是评估范围。自动化测试验证基础行为，但尚未执行更广泛的用户测试和检索质量测量。

## 5.7 对课程目标的达成

项目通过一个可工作的全栈应用展示了软件中心系统创建能力。它应用了当前软件设计实践，例如分层架构、API
契约、组件化前端设计、本地优先数据处理和检索增强生成。它还通过结构化仓库、里程碑和职责分离展示了项目管理和团队合作原则。

概念验证在技术上可信。它实现了提案中的核心功能，并留下现实可行的扩展路径。最终设计与课程目标一致，即构建软件中心系统，而不仅仅是描述系统。

## 5.8 项目过程反思

项目过程中的主要经验是应尽早处理集成风险。在 RAG
系统中，文档解析、检索、提示词构造、LLM
调用和前端展示相互依赖。过晚集成这些部分会使后期调试困难。尽早构建最小端到端路径使后续变更更安全。

第二个经验是本地优先系统仍然存在外部依赖风险。即使文档是本地的，OCR
引擎、模型 API
和机器学习包仍可能引入设置复杂度。项目通过回退路径和清晰错误信息应对这一点。

第三个经验是配置简单性很重要。中间 API 网关配置带来了 base URL
和模型名的不确定性。转向 DeepSeek 官方 API 变量使系统更容易解释和运行。

第四个经验是引用需要从一开始就设计。如果页码和文档元数据没有通过切块和检索过程保留下来，后续做来源归因会很困难。已实现数据模型从早期就保留了这些元数据。

## 5.9 非功能质量评估

可用性由聚焦的文档页和聊天页、可见处理状态以及可展开引用支持，但缺少集成 PDF/图片查看器仍然限制了来源验证。可维护性来自路由、服务、模型、API 客户端和 UI 组件之间的分离。隐私性由本地文件存储、本地检索以及在配置 DeepSeek 后只外发选中片段的机制支持。

可靠性由明确错误状态和回退路径支撑，能够处理缺少 API 密钥、不支持格式、OCR 失败和 LLM 失败等情况。对于小规模概念验证数据，性能是足够的；但同步处理和回退扫描在较大集合上需要后台任务和持久化向量索引。可移植性较好，因为系统使用常见 Web 技术和本地数据库，但 Tesseract 与可选机器学习包仍是系统级依赖。

## 5.10 风险分析

主要技术风险是依赖可用性、检索准确性和 LLM 可靠性。OCR 依赖 Tesseract，语义检索依赖可选机器学习包，生成回答仍可能不完整或表述不准确。项目通过清晰错误、回退检索、保守提示词和可见引用来缓解这些风险。

项目还存在隐私和范围风险。用户可能误解本地优先，认为内容永远不会离开本机；但当配置 API 密钥后，选中的片段会发送给 DeepSeek。因此系统需要清楚说明数据流。范围膨胀也是风险，因为文档查看器、知识图谱、协同、移动端和云部署都很有吸引力，但不属于已批准的项目范围。

## 5.11 伦理与专业考虑

系统处理个人文档，因此透明性和用户控制非常重要。原始文件留在本地，检索在本地执行，外部生成是可选项。启用 DeepSeek 时，只有检索片段会被发送，但这些片段仍可能包含敏感内容。因此用户应在启用生成回答前理解数据流。

系统也不能夸大回答可靠性。生成回答有帮助，但并非权威来源，尤其是在学术、医疗、法律或财务文档场景中。引用和来源预览的作用是让用户能够验证关键说法。

## 5.12 总体成果

总体而言，项目达成了主要目标。它产出了一个可工作的本地优先多模态个人知识库
QA
概念验证。系统连贯、可演示，并与拟定范围一致。它实现了文档上传、本地处理、切块、检索、回答生成和引用展示。

项目也展示了设计成熟度。它包含回退路径、清晰状态值、模块化服务、类型化前端
API 调用和文档。它没有隐藏限制，而是明确指出限制并识别现实未来改进。

因此，最终成果对于预期评估语境来说是成功的。它不是完成品，但它是继续开发的强基础，并且是软件设计与实现能力的可信展示。

# 6. 结论与未来工作

## 6.1 结论

多模态个人知识库问答系统成功展示了拟定的本地优先 RAG
工作流。它允许用户上传个人文档，在本地处理它们，检索相关文本块，提出自然语言问题，并查看带引用的回答。实现使用清晰的分层架构，分离前端和后端职责，本地存储数据，并在配置后集成
DeepSeek 官方 API 生成回答。

系统不是完整商业知识管理产品，但它是可信的概念验证。它实现了提案中的核心功能，并符合课程要求，即展示软件设计和实现能力。其最强特征是端到端工作流、本地优先数据处理、结构化引用、模块化后端服务和优雅回退行为。

项目也揭示了现实限制。OCR
依赖外部安装，检索质量需要更广泛评估，大型文档处理应移到后台任务，来源验证会受益于更丰富的文档查看器。这些限制并不削弱概念验证；相反，它们指出了清晰的下一步。

## 6.2 未来工作

未来工作应集中在延展当前设计、而不是改变项目范围的现实改进上。第一优先级是后台处理，使大型 PDF 和 OCR 密集型图片不会阻塞上传请求。第二是更丰富的引用查看器，让用户可以打开引用对应的具体 PDF 页或图片区域。第三是更强的检索能力，包括完整语义检索设置、词法-语义混合排序、重排序和基准问题集。

进一步扩展包括加密本地存储，支持 Markdown、DOCX、PPTX、CSV 和 HTML 等更多格式，改进用于学习摘要或文档比较的提示词模板，开展代表性用户可用性测试，以及为非技术用户提供打包启动器。这些改进都自然延续当前概念验证，同时保留本地优先原则。
